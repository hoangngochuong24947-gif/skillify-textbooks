# 从书生成 Skills 头脑风暴文档

## 1. 为什么这个方向值得做

很多书的真正价值不在于结论，而在于作者如何思考、如何拆问题、如何组织判断。如果这些“思考方式”只能停留在阅读者脑中，它们就很难被复用、很难被迁移、也很难被 agent 软件调用。

这个项目真正想做的是：

- 把一本好书里的思维结构显式化。
- 把零散方法论模块化。
- 把模块变成可调用 skill，而不是普通读书笔记。
- 把“用这本书的思维思考”从一句口号变成一套可执行流程。

## 2. 这不是普通读书笔记

普通读书笔记关注“书讲了什么”，本项目关注“如何把书变成会工作的东西”。

具体差异如下：

| 普通笔记 | 本项目 |
| --- | --- |
| 摘要与金句 | 方法卡与路由规则 |
| 线性阅读记录 | 面向任务的技能编排 |
| 人看完即可 | agent 可继续调用 |
| 偏内容保存 | 偏推理资产沉淀 |

## 3. 可以抽出来的三层价值

### 3.1 作者层

- 作者如何观察问题。
- 作者默认站在哪些视角上判断。
- 作者如何区分表象、结构、因果和约束。
- 作者使用了哪些高频概念对现实进行切分。

### 3.2 方法层

- 显性方法：作者直接给出的步骤、模型、框架。
- 隐性方法：作者没有命名，但反复使用的分析套路。
- 组合方法：多个零散观点拼起来才能形成完整技能。

### 3.3 任务层

- 什么问题适合调用这本书。
- 什么问题只能调用部分方法，而不能整本照搬。
- 什么问题需要多个方法联动。
- 什么问题不适合使用本书视角，应该明确拒绝。

## 4. 项目最有力量的地方

### 4.1 从“内容”转向“执行能力”

如果最终只是得到一堆解释性文字，价值有限；如果最终能得到一个会判断、会挑方法、会调用子模块的 skill，价值就完全不同。

### 4.2 从“单人阅读收益”转向“多人可复用资产”

你读过一本书后的理解，通常只服务你自己；但如果把这套理解变成可调用的 skill，其他人可以直接借用你的抽取成果。

### 4.3 从“把书喂给模型”转向“把书组织成方法论工厂”

NotebookLM 在这里不是摘要工具，而是知识萃取引擎。真正的产物不是问答，而是可编排的结构化资产。

## 5. 需要特别避免的坑

### 5.1 只做摘要，不做方法卡

这是最容易滑过去的方向。只要没有沉淀成 `MethodCard`，最终就无法路由、无法委派、也无法重用。

### 5.2 把“作者观点”误当作“通用方法”

有些书适合提供视角，不一定适合提供流程。第一版需要允许“这本书适合做 worldview skill，不适合做 full-stack methodology skill”的判断。

### 5.3 没有证据约束，导致强行脑补

书中的隐性方法确实要靠归纳，但所有归纳都必须保留证据指向，否则 skill 会快速漂移。

### 5.4 让 subagent 自由生长

如果 subagent 的输入输出不固定，很快就会失控。第一版要压缩自由度，先做稳定结构。

## 6. 可以延展出的产品形态

### 6.1 单书 Skill

适合强方法论图书，例如决策、战略、写作、投资、沟通、学习等主题。

### 6.2 书系 Skill

同一作者、同一流派或同一主题的多本书，可以在后续版本做融合对比。

### 6.3 任务型 Skill 市场

最终甚至不一定按书组织，而是按任务组织。书只是来源，最终呈现给用户的是：

- 产品战略分析 skill
- 投资判断 skill
- 学习加速 skill
- 研究设计 skill

## 7. Claude Code / OpenClaw 视角下的价值

这类宿主本来就擅长拆任务、调工具、产文档，但它们缺的是“这本书的方法论地图”。这个项目相当于给宿主补上一层深度认知插件。

最终用户体验应接近：

1. 用户提出问题。
2. master skill 判断能否使用某本书的思维。
3. 如果适合，就选择方法模块。
4. 再选择需要的 subagents。
5. 最后输出分析结果与中间文档。

## 8. 一套更成熟的命名方式

当前“从书生成 skills 的 skills”这个表达是对的，但对外可以更清楚：

- `book-to-skills`
- `methodology-extractor`
- `book-thinking-router`
- `author-mindset-skillforge`

如果后面做产品名，建议“book-to-skills”保留为工程名，外部展示名再单独打磨。

## 9. 建议的近期路线

### 9.1 第一阶段

- 固化文档结构。
- 固化问题包。
- 固化 subagent 角色。
- 固化 `MethodCard` 和 `RoutingDecision` 模式。

### 9.2 第二阶段

- 用一本典型书做完整试跑。
- 找出问题包里提问不稳的地方。
- 调整哪些结论能自动化，哪些需要人工复核。

### 9.3 第三阶段

- 让 Claude Code 能稳定读取这套 skill。
- 做到对用户请求自动判断调用哪套方法。
- 再考虑接入更多宿主和更多书。

## 10. 这轮产物最应该服务什么

这轮不是为了把功能做满，而是为了把你脑中的发散想法压缩成一个非常清晰的骨架：

- 看得懂。
- 能扩展。
- 能交给 Claude Code 执行。
- 后续能拿去做图、做 demo、做第一本书的试跑。
